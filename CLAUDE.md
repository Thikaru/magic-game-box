# Daily Games リポジトリ規約

このリポジトリは「毎日1つ新しいミニゲームが増える」静的サイトです。
GitHub Pages で公開され、Claude Code の Routines によって自動更新されます。
以下の規約は新規作成・修正のどちらのルーチンでも必ず守ること。

## ディレクトリ構成

```
/
├── index.html            # トップページ(本日のゲーム)※基本的に編集しない
├── archive.html          # 過去ゲーム一覧          ※基本的に編集しない
├── assets/style.css      # サイト共通スタイル      ※基本的に編集しない
├── games.json            # ゲームレジストリ(新規作成時に必ず追記)
├── routines/
│   └── games-log.md      # 制作記録(ジャンル・コンセプトの重複防止用、新規作成時に必ず追記)
└── games/
    └── YYYY-MM-DD-<slug>/   # ゲームごとに1フォルダ。フォルダ外に一切影響を与えない
        ├── index.html       # ゲーム本体(このファイルだけで完結してもよい)
        ├── game.js          # ロジック(分割する場合)
        └── meta.json        # メタ情報
```

## ゲームフォルダの命名

- `games/YYYY-MM-DD-<slug>/`。slug は英小文字とハイフンのみ(例: `2026-07-08-space-shooter`)
- 日付は実行日(Asia/Tokyo)

## games.json のスキーマ

新しいゲームを作ったら `games` 配列の**先頭**に追加する(先頭 = 最新 = 本日のゲーム)。

```json
{
  "games": [
    {
      "id": "2026-07-07-block-breaker",
      "title": "ブロックくずし",
      "genre": "アクション",
      "date": "2026-07-07",
      "description": "パドルでボールを打ち返してブロックを全部壊そう。",
      "path": "games/2026-07-07-block-breaker/"
    }
  ]
}
```

`meta.json` にも同じ内容を1オブジェクトで保存する。

## ゲームの必須要件

1. **説明から開始**: ロード直後はゲームを開始せず、タイトル・遊び方・操作方法を
   表示した説明画面(オーバーレイ)を出す。「スタート」ボタンで開始する。
2. **何度でもやり直せる**: ゲームオーバー/クリア画面に「もういちど」ボタンを置き、
   ページを再読み込みせずに最初から遊べるようにする。
3. **ヘッダーにトップへ戻るリンク**: `../../index.html` へのリンクを置く。
4. **完全に自己完結**: 外部ライブラリ・CDN・画像ファイルを使わない。
   Vanilla JS + Canvas または DOM のみ。絵は Canvas 描画・絵文字・CSS で表現する。
5. **PC とスマホの両対応**: キーボード操作に加え、タッチ/クリック操作を必ず実装する。
   - タップ・ドラッグ・スワイプで直接遊べるゲーム(タップ進行、パドルのドラッグ移動など)は
     それだけでよい
   - キー入力が前提のゲーム(十字キー移動・ジャンプボタンなど)は、画面内に**仮想コントローラ**
     (十字キー/アクションボタン)を実装する。表示ルール:
     - `matchMedia('(pointer: coarse)').matches` が true ならデフォルトで表示する
     - さらに「🎮 コントローラ」の表示/非表示切替ボタンを常設する(自動判定が外れた時の保険)
   - 幅 375px の縦画面で横スクロールが発生しないこと。タップ対象のボタンは高さ 44px 以上にすること
6. **1ファイル or 最小構成**: `index.html`(+必要なら `game.js`)に収める。
7. **サウンド(音が体験の核となるジャンルの場合)**: リズム/音ゲーなど「音が鳴ること」
   自体がゲーム性の一部であるジャンルでは、実際に効果音・BGM が鳴るようにすること。
   - 外部の音楽・効果音ファイルを CDN 等から読み込むのは禁止(必須要件4に反する)。
   - 基本は Web Audio API(`OscillatorNode` / `AudioBuffer` でのビープ音・ノイズ生成)
     による自前生成の音を使う。著作権の心配が一切ない。
   - パブリックドメイン/CC0 など著作権が明確にない(フリーの)音源ファイルが
     手元にあり、それをゲームフォルダ内にローカル保存して使う場合のみ、
     ファイルとして含めてよい。その際はライセンスと出典を `meta.json` か
     フォルダ内のコメントに明記すること。
   - `AudioContext` の生成・再生は必ずユーザー操作(スタート/リトライボタンの
     クリックなど)の中で行うこと(ブラウザの自動再生制限のため、操作なしで
     鳴らそうとすると音が出ない)。

## ジャンルのローテーション

`games.json` を読み、**前日(直近1件)と同じジャンル**にならないようにする。
何日も連続で同じジャンルにならなければよく、直近7件全体との重複は気にしなくてよい。
ジャンル例:
アクション / パズル / シューティング / レース / リズム / クイズ / タイピング /
反射神経 / 記憶 / カード / 育成 / ローグライク / スポーツ / 脱出 など。

ジャンル内でも毎回ひとひねり入れ、過去作の焼き直しにしないこと。作る前に
`routines/games-log.md`(制作記録)に目を通し、既存作品と同じコンセプト・
仕掛け(例:「特定の的だけは避けてタップする」といった逆転ギミックなど)を
繰り返していないか確認すること。

**ジャンルの組み合わせ**: 単一ジャンルだけでなく、2つ以上のジャンルを
組み合わせたゲームも積極的に作ってよい(例: 「パズル×レース」「カード×育成」
「タイピング×クイズ」など)。組み合わせる場合、`genre` フィールドは
`"パズル×レース"` のように `×` で連結して記載する。

## 制作記録(routines/games-log.md)

過去に作ったゲームのジャンル・コンセプト・ひとひねりポイントを一覧にした
制作記録を `routines/games-log.md` に保持する。新しいゲームを作ったときは、
`games.json` への追記に合わせてこのファイルにも1行(日付・ID・ジャンル・
コンセプトの要約)を追記すること。ジャンルや仕掛けの重複を避けるための
参照資料なので、新規作成前に必ず目を通し、作成後は必ず更新する。

## 品質チェック(コミット前に必ず)

- `python3 -m json.tool games.json` で JSON が壊れていないこと
- `node --check games/<id>/game.js`(game.js がある場合)で構文エラーがないこと
- 説明画面 → プレイ → 終了 → もういちど、の一連の流れが成立するコードであること
- 幅 375px の縦画面で横スクロールが発生せず、タッチ操作(または仮想コントローラ)だけで
  最後まで遊べること

## Git 運用

- ルーチンは `claude/` プレフィックスのブランチにプッシュし、PR を作成する
- PR タイトルは新規作成なら `game: YYYY-MM-DD <タイトル>`、修正なら `fix: <対象ID> <概要>`
- `claude/**` ブランチの PR は GitHub Actions が自動マージする(.github/workflows/automerge.yml)
